# 2026-05-06

## Syntax highlighting + draggable tabs

### Changes

- **CodeEditor.tsx** — registered `syntaxHighlighting(defaultHighlightStyle, { fallback: true })` as a non-compartmented extension. The `fallback: true` flag means it only activates when no other highlighter is loaded, so it covers light mode while `oneDark` continues to handle dark mode (oneDark bundles its own highlighter). Single-line addition; no theme-coupling, no conditional logic in the reconfigure lifecycle.
- **TabBar.tsx** — refactored to wrap each `TabItem` in a `relative` container and added native HTML5 drag-and-drop handlers (`draggable`, `onDragStart`, `onDragOver`, `onDragLeave`, `onDrop`, `onDragEnd`). Wires up the pre-existing `reorderTabs(fromIndex, toIndex)` store action that had no UI consumer. A 2px absolute-positioned `--accent`-colored bar marks the drop target during hover (no layout shift). Local `dragOverIndex` state — kept out of Zustand because it's transient UI scoped to one render cycle.
- **Tests** — added 2 reorder interaction tests in `TabBar.test.tsx` (drag-to-position, drop-on-source no-op) and 1 token-span test in `CodeEditor.test.tsx`. All 186 tests pass (was 183).
- **ARCHITECTURE.md** — updated Editor Components section.

### Verification

- `pnpm test`: 186/186 pass.
- `pnpm dev` + browser at `http://localhost:1420`: switched to light mode, confirmed JS keywords (`const`, `function`) render in distinct colors via `defaultHighlightStyle`. Programmatically simulated drag from tab[0] → tab[2] using `new DragEvent(...)` with a real `DataTransfer`; the store updated and the visible tab order reflected the reorder. Active-tab underline survives a reorder.

### Edge cases handled

- `reorderTabs` already validates indices (out-of-range, identical from/to, single-tab arrays); no duplicate guards added.
- `Number.parseInt` on the dragged-data string + `Number.isNaN` guard so external drops (e.g. files dragged from Finder) don't call `reorderTabs(NaN, ...)`.
- `onDragOver` calls `e.preventDefault()` (required — without it, `onDrop` never fires; common HTML5 DnD gotcha).

### Notes for future me

- jsdom doesn't implement `DataTransfer` — Vitest tests stub it via `fireEvent.drop(el, { dataTransfer: { getData: () => '0' } })`. In the live browser, real `DragEvent` + `new DataTransfer()` works fine.

For UI verification, injecting tabs directly via `await import('/src/store/editorStore.ts')` + `useEditorStore.getState().openFile(...)` works because Zustand stores are module singletons and Vite serves source modules at predictable paths in dev.
